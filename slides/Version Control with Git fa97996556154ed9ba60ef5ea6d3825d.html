What is version control?

  • Version control is the practice of tracking and managing changes to software code.
  • Keeps track of every modification to the code in a special kind of database.
  • Enables multiple people to work on a project simultaneously without having to worry about synchronisation.
  • If a mistake is made, developers can turn back the clock and compare earlier versions of the code to help fix the mistake while minimizing disruption to all team members.

What not to do on your projects:

Ever created a million final final definitely final .... versions of your project files / folders?

Ever used IM platforms / email to share code while collaborating on projects?

Consider this scenario

  • You have a CS101 programming assignment due at midnight and you have it ready for submission. It is too close to the deadline now.
  • While testing for one last time, you discover a minor bug and decide to quickly fix it.
  • While attempting to fix the bug, you accidentally break the working code and end up in a big mess
  • You can't recall what you changed and what you didn't.
  • It's 23:57 PM
  • Ctrl + Z can't help you

Distributed Version Control

  • Version control system is a system that records changes to a file or set of files over time so that you can recall specific versions later
  • Distributed means that there is no main server and all of the full history of the project is available once you cloned the project
  • Changes to files are tracked between computers. For example, my workstation and yours. In the beginning, this required specific coordination strategies to maintain consistency in projects, so all the developers could keep track of what was happening to files at any given time

What is Git?

  • Open source project originally developed in 2005 by Linus Torvalds
  • A command line utility
  • A distributed version control system - DCVS
  • Available on Windows, macOS and Linux

More about Git

  • You can imagine git as something that sits on top of your file system and manipulates files
  • This "something" is a tree structure where each commit creates a new node in that tree
  • Nearly all git commands actually serve to navigate on this tree and to manipulate it accordingly

Basic Git Workflow

  • Modify files in your working directory
  • Stage files, adding snapshots of them to your staging area
  • Commit, which takes the files in the staging area and stores that snapshot permanently to your Git directory.

Basics

  • Setup and configuration (Do once in your entire git setup in your machine)
    git config --global user.name "Your Name" # Change the name you commit as by default
git config --global user.email "your_email@whatever.com" # Change the email id you commit from by default
git config --global core.editor vim   # Change your default editor (can be anything, eg code, vim, emacs, notepad, subl etc,
																			# any editor that can be opened from the command line)
  • Creating a project (Do this once per project)
    git init
  • Checking Status of the Repo (Do each time you want to check status of staging area and working directory)
    git status
  • Staging and Un-staging
    git add [file] # Add the file to staging area
git add -A # Add all changes to files and new files and deletion of files to staging area
git add . # Add all changes to files and new files to staging area
git add --update # Add all changes to files to staging area
  • Committing
    git commit # Commit all changes in staging area. Opens up default text editor in which you can 
					 # write commit message subject and also optional commit message body
git commit -m 'commit message' # Commit using the commit message subject
git commit -a # Commit all changes in already staged files directly without needing to stage the changes
git commit -am 'commit message' # Use both the a and m flag together
  • git diff
    git diff [file]
  • How to see which branches are there
    git branch
  • Adding a branch
    git branch [branch-name]
  • Switching to a branch
    git checkout [branch-name]
  • Logs and history
    git log
git log --oneline
git log --pretty=format:'%h %ad | %s%d [%an]' --graph --date=short
git log --pretty=format:'%h %ad | %s%d [%an]' --graph --date=short --all
  • Checking out an older commit
    git checkout [commit-hash]
  • Create a new branch from an older commit
    git checkout [old-commit-hash] #First go to older commit
git checkout -b [new-branch-name] # Create and switch to a new branch from here

Playing with Branches

  • Merging changes from master
    git checkout [branch] # checkout branch
git merge master # select which branch you want to merge into the current branch
# here we are merging master into current branch

# To merge sub into master do the following:
git checkout master
git merge sub
  • Conflicts and resolution
  • Rebasing
    git rebase [branch] # merges all changes from the [branch] and removes all traces that the branch existed
# all commits in history show the [branch] commits as master commits after rebasing
  • Undoing unstaged changes
    git checkout [file] # Undoes any change in the file since last commit
  • Undoing staged changes
    # First unstage
git restore --staged [file]
# or
git reset HEAD [file]

# now undo changes
git checkout [file]
  • Undoing commits (revert)
    git revert HEAD # Revert the last commit (where HEAD is at)
git revert [commit-hash] # Revert back to an older commit
  • Undoing commits (reset)
    git reset --hard [hash]
git reset --soft [hash]
git reset --mixed [hash]
# Look up more options in the documentation
  • Amending last commit
    git commit --amend -m [message]
# git commit --amend will also work of course, it will open up the default text editor

Remote Repos and GitHub

  • Git vs Github

    Git is a version control software while GitHub is designed as a Git repository hosting service. It enables people to collaborate through the Internet and keep track of each other's changes.

  • Remote repos
  • Cloning remote repos
    git clone [remote-link]
  • Master vs Main (GitHub drama)
  • Adding remote to a local repo
    git checkout master # switch to master if already not on master
git remote add origin [remote-link] # add the remote named origin to local repo
git push -u origin master # set the current branch master to track the remote branch master and push
  • New remote repository on GitHub:
  • Push / Pull from remote repo
    git pull
git push
# always pull before pushing
# can directly do git push if you already know you have the changes from remote
  • Changing upstream branch
    git branch [local_branch_name] --set-upstream-to origin [remote_branch_name]
# It is preferred to keep local and remote branch name to be same
  • Collaboration
  • Adding SSH keys (to prevent entering username and password every time)
